# resumellmatch MVP Planning

## Overview

This document outlines the MVP (Minimum Viable Product) for resumellmatch based on Pareto principle: the 20% of features that deliver 80% of user value. Focus is on reliability, speed, and core functionality. All "nice-to-have" features are deferred to post-MVP phases.

> **Note:** the initial storage layer will use SQLAlchemy with an SQLite database to keep setup simple; configuration must allow switching to PostgreSQL later when richer LLM-related data (embeddings, vectors) require it.

---

## Major Part 1: Data Model & Storage Layer

**Purpose**: Reliable resume & star data model that prevents bugs, data loss, and supports fast queries.

### Minor Part 1.1: Database Schema Setup

- **Status**: Not Started
- **Description**: Use **SQLAlchemy ORM** with a lightweight SQLite database for the initial MVP. The schema should be defined in SQLAlchemy models and migrations generated by Alembic. The configuration must allow switching to PostgreSQL (with pgvector extension) once the project grows and vector queries are needed.
- **Items to Modify**:
  - Define all core models (`User`, `StarMetadata`, `StarEntry`, `StaticSectionEntry`, `Resume`, `ResumeSection`, `ResumeSectionEntry`, `JobDescription`, `JobParsed`, `Match`, etc.) using SQLAlchemy declarative base
  - Create Alembic migration files for each model so the same migrations work against both SQLite (dev/test) and PostgreSQL (prod)
  - For PostgreSQL target later, add pgvector extension support in a migration and include `Vector` columns where needed (e.g. in an `embeddings` table) – these fields should be optional or omitted on SQLite.
  - Create indexes on `user_id`, `resume_id`, `jd_id` (and later on vector columns) for query performance
  - Set up foreign key constraints and cascading deletes in the models/migrations
- **Definition of Done**:
  - All tables can be created via Alembic in an SQLite database and later in PostgreSQL
  - Alembic is installed and tracked, and a migration workflow is documented
  - Database connections (SQLite during development) are testable from CLI and code can easily switch to a PostgreSQL URL

### Minor Part 1.2: Repository Layer (CRUD Operations)

- **Status**: Not Started
- **Description**: Implement CRUD operations via repositories (resume_repo.py, jd_repo.py, star_repo.py, match_repo.py).
- **Items to Modify**:
  - Create `storage/repositories/resume_repo.py` with fetch, create, update, delete for resumes and linked sections
  - Create `storage/repositories/star_repo.py` for managing STAR entries and metadata
  - Create `storage/repositories/jd_repo.py` for job descriptions and parsed data
  - Create `storage/repositories/match_repo.py` for storing and querying match scores
  - Implement transaction handling for multi-step operations (e.g., resume + sections in one call)
- **Definition of Done**:
  - All repositories have tested CRUD methods
  - Transactional integrity verified (e.g., deleting resume also deletes child records)
  - Fast queries (< 100ms for common operations) via proper indexing

### Minor Part 1.3: Database Connection Pool

- **Status**: Not Started
- **Description**: Configure SQLAlchemy's engine with a connection pool so that the same code works with SQLite during development and PostgreSQL in production. Pool settings should be adjustable via environment variables or config.
- **Items to Modify**:
  - Implement `storage/db.py` that creates an SQLAlchemy `Engine` and `SessionLocal` with a configurable connection URL
  - Expose parameters for pool size, timeout, and idle timeout in the configuration
  - Add logging for connection health and SQL queries when verbose/debug mode is enabled
- **Definition of Done**:
  - SQLAlchemy engine is initialized at app startup with pool settings
  - Connections are reused and released properly in both SQLite and PostgreSQL
  - No connection leaks under concurrent CLI commands

## Major Part 2: Configuration & Settings

**Purpose**: Centralize application configuration so logging levels, database connections, LLM host/keys, and ranking parameters can be changed without code edits. Settings should be loaded from environment variables or a config file with sensible defaults, and the same system used throughout all components.

### Minor Part 2.1: Configuration Framework

- **Status**: Not Started
- **Description**: Build a single `config` module (e.g. using Pydantic's `BaseSettings` or `dynaconf`) that reads from environment variables and/or a `.env` file. The module should expose typed attributes with defaults and validation.
- **Items to Modify**:
  - Create `config/settings.py` implementing the settings class
  - Support keys for `LOG_LEVEL`, `DATABASE_URL`, `OLLAMA_HOST`, `OLLAMA_API_KEY`, `RANK_WEIGHT_KEYWORDS`, etc.
  - Allow overriding via command‑line flags in the CLI entry point
  - Provide a helper to reload or access settings globally
- **Definition of Done**:
  - Settings object is instantiated at program start and accessible from all modules
  - Invalid configuration values raise clear errors
  - Switching `DATABASE_URL` toggles between SQLite and PostgreSQL seamlessly

### Minor Part 2.2: Logging Configuration

- **Status**: Not Started
- **Description**: Configure Python's logging module using the log level from settings and optionally direct output to console/file.
- **Items to Modify**:
  - Add `logging_setup.py` to apply configuration at startup
  - Respect `LOG_LEVEL` and `LOG_FILE` settings
  - Add verbose/debug option in CLI to temporarily override log level
- **Definition of Done**:
  - Logs respect configured level and format
  - CLI verbose flag works correctly
  - Logging is available in libraries (SQLAlchemy, Ollama client) via configuration

### Minor Part 2.3: Ranking & Algorithm Parameters

- **Status**: Not Started
- **Description**: Expose weights, thresholds, and other tunables used by the matching algorithm in the configuration system so they can be adjusted without code changes.
- **Items to Modify**:
  - Add settings like `KEYWORD_WEIGHT`, `SEMANTIC_WEIGHT`, `SENIORITY_THRESHOLD`, etc.
  - Ensure `core/ranker.py` reads these values from the settings object
  - Add simple documentation for each parameter
- **Definition of Done**:
  - Algorithm behaviour changes when parameters are tweaked via env vars
  - Values are validated for sensible ranges

### Minor Part 2.4: CLI Integration & Global Options

- **Status**: Not Started
- **Description**: Make CLI commands aware of configuration and allow overriding certain settings via command-line arguments.
- **Items to Modify**:
  - In `cli/main.py`, add global flags `--db-url`, `--log-level`, `--ollama-host` that override corresponding settings
  - Pass settings instance into command handlers
  - Document CLI configuration options in help text
- **Definition of Done**:
  - Users can start the CLI with custom settings without editing environment files
  - Flags properly override settings and are logged on startup

## Major Part 3: LLM Integration & Prompt Management

**Purpose**: Reliable LLM calls to Ollama that handle errors, respect token limits, and enable prompt versioning.

### Minor Part 3.1: Ollama Client

- **Status**: Not Started
- **Description**: Create robust Ollama client with error handling, retries, and token counting.
- **Items to Modify**:
  - Implement `llm/ollama_client.py` with methods for text generation and embeddings
  - Add exponential backoff for failed requests
  - Implement token counter to respect context window limits
  - Add logging for all LLM calls (model, tokens, latency)
- **Definition of Done**:
  - LLM calls succeed reliably in happy path
  - Timeouts and errors are handled gracefully with user-friendly messages
  - Token count is tracked and logged for cost estimation

### Minor Part 3.2: Prompt Management & Versioning

- **Status**: Not Started
- **Description**: Define core prompts with versioning to enable A/B testing and iterative improvements.
- **Items to Modify**:
  - Create prompt files in `llm/prompts/`:
    - `extract_keywords.txt` - extract skills and keywords from JD
    - `rank_resume.txt` - score a resume against a JD
    - `star_selection.txt` - identify which STAR entries match a JD
    - `star_rewrite.txt` - rewrite STAR entry to match JD context
  - Add prompt versioning logic (e.g., v1.0, v1.1)
  - Create prompt templates with variable substitution (e.g., {{jd_content}}, {{resume_content}})
- **Definition of Done**:
  - Each core prompt is documented with inputs, outputs, and example usage
  - Prompts can be updated without code changes
  - Prompt version is logged with every LLM call

### Minor Part 3.3: Embedding Generation & Caching

- **Status**: Not Started
- **Description**: Generate embeddings for JDs and resume content via Ollama, with caching to reduce recompute.
- **Items to Modify**:
  - Implement `llm/embeddings.py` with methods to generate vector embeddings
  - Add `embeddings` table to store embeddings with source type, content, and vector
  - Cache embeddings in DB to avoid re-embedding identical content
  - Implement embedding invalidation when source content changes
- **Definition of Done**:
  - Embeddings are generated on first call and cached
  - Cache hit rates are logged
  - Vector dimension matches database schema (1536 or Ollama model output)

## Major Part 5: Ranking & Matching Algorithm

**Purpose**: Fast, deterministic scoring of resumes against job descriptions with explainability.

### Minor Part 5.1: Ranking Algorithm

- **Status**: Not Started
- **Description**: Implement core ranking logic that scores a resume against a JD with multiple signals.
- **Items to Modify**:
  - Implement `core/ranker.py` with scoring method that combines:
    - Keyword overlap (exact matches between resume and JD)
    - Semantic similarity (cosine distance between embeddings)
    - Skill coverage (required vs. present skills)
    - Seniority alignment (years of experience match)
  - Normalize scores to 0–100 scale
  - Store match result in `matches` table with score and features
- **Definition of Done**:
  - Ranking produces consistent scores for the same resume-JD pair
  - Score differences are explainable (e.g., "missing 3 required skills")
  - Ranking is fast (< 500ms per match)

### Minor Part 5.2: Match Repository & Querying

- **Status**: Not Started
- **Description**: Store and query match results efficiently for ranking and comparison.
- **Items to Modify**:
  - Extend `storage/repositories/match_repo.py` with:
    - `create_match(resume_id, jd_id, score, features)` to save match
    - `get_top_matches(jd_id, limit, min_score)` to fetch best resumes for a JD
    - `get_matching_jds(resume_id, limit, min_score)` to find best JDs for a resume
  - Implement indexes on (resume_id, jd_id) for fast lookups
- **Definition of Done**:
  - Matches are persisted in database
  - Querying top matches returns results in < 100ms
  - Duplicate matches (same resume-JD pair) are prevented

### Minor Part 5.3: Match Explanation via Features

- **Status**: Not Started
- **Description**: Store and display match reasoning (what skills match, what's missing, etc.).
- **Items to Modify**:
  - Extend `Match` model to include `match_features` JSONB field with:
    - `cosine_similarity`: semantic similarity score
    - `keyword_overlap`: list of matched keywords
    - `skill_coverage`: required skills found vs. missing
    - `seniority_match`: true/false if experience level aligns
  - Add LLM-generated `llm_analysis` field for natural language explanation (optional for MVP)
- **Definition of Done**:
  - Match features are computed and stored
  - Features enable users to understand why a resume scored high/low
  - Features are queryable for debugging and analysis

## Major Part 6: Semantic Search & Embeddings

**Purpose**: Use pgvector to find relevant STAR entries and resume content for a given job description.

### Minor Part 6.1: pgvector Setup & Integration

- **Status**: Not Started
- **Description**: Enable pgvector extension in PostgreSQL and create vector indexes.
- **Items to Modify**:
  - Create migration to enable `pgvector` extension
  - Add `embedding vector(1536)` column to `embeddings` table
  - Create index on embeddings: `CREATE INDEX ON embeddings USING ivfflat (embedding vector_cosine_ops)`
  - Update `storage/db.py` to use SQLAlchemy-provided vector type
- **Definition of Done**:
  - pgvector extension is enabled in PostgreSQL
  - Vector indexes are created and usable
  - Vector similarity queries complete in < 500ms for typical dataset sizes

### Minor Part 6.2: Embedding Storage & Retrieval

- **Status**: Not Started
- **Description**: Store embeddings for all resume sections, STAR entries, and JD content; enable fast retrieval.
- **Items to Modify**:
  - Extend `embeddings` table schema with proper source type classification
  - Implement methods to store embeddings with metadata (source_id, source_type, created_at)
  - Implement similarity search: `find_similar_entries(embedding_vector, source_type, limit)` in `storage/repositories/`
  - Update embedding lifecycle: regenerate when source content changes
- **Definition of Done**:
  - Embeddings are generated and stored for all content
  - Similarity search returns top-k relevant entries quickly
  - Embeddings are regenerated when source content is updated

### Minor Part 6.3: Semantic Matching for STAR Selection

- **Status**: Not Started
- **Description**: Use embeddings to automatically identify which STAR entries are most relevant for a given JD.
- **Items to Modify**:
  - Implement `core/star_manager.py` with method `find_relevant_stars(jd_embedding, user_id, limit)`
  - Use vector similarity search to rank STAR entries by relevance
  - Cache results to avoid recomputing for same JD/resume combination
- **Definition of Done**:
  - STAR entries can be ranked by semantic relevance to a JD
  - Top STAR entries are returned in < 1 second
  - Semantic relevance correlates with manual assessment (spot check)
